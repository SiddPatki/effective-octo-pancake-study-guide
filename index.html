A

GPU + AI Infrastructure Basics

The most important module — your real gap
Why it matters
Every other topic points back to one question: is the GPU doing useful work, and does it have what it needs? You can't debug "slow inference" or "out-of-memory" without understanding what a GPU actually does.

Concepts

A1 · GPU vs CPU

A CPU is a small team of very smart generalists: a handful of powerful cores that each handle complicated, different tasks fast — great for logic and running the OS. A GPU is a stadium full of simple workers: thousands of small cores that all do the same simple math at the same time (parallelism).

AI needs this because a neural network is, underneath, an enormous pile of multiplications and additions on grids of numbers (matrices) — exactly the "same operation, millions of times" work GPUs were born for. A GPU does it 10–100× faster than a CPU.

🧠
Mental model: CPU = a few geniuses; GPU = a massive crowd doing identical math in unison.

A2 · CUDA — how software talks to the GPU

CUDA is NVIDIA's software layer (toolkit + drivers) that lets programs send work to NVIDIA GPUs. A "CUDA error" in logs almost always means "something went wrong while the GPU was working" — most often running out of memory.

Two pieces you'll see: the GPU driver (low-level software on the node so the OS can see/control the GPU — missing/wrong version means the GPU effectively "doesn't exist"), and the CUDA toolkit/runtime (libraries the app uses — a version mismatch crashes things on startup). So when a GPU "isn't working," first check driver/CUDA visible and healthy? before blaming the model.

A3 · VRAM vs RAM — the concept behind most incidents

The CPU uses RAM. A GPU has its own separate memory called VRAM. They are not the same pool. To run a model on a GPU, its weights must physically fit inside VRAM (a GPU might have 24, 40, 80, or 141 GB). Exceed it and you get an out-of-memory (OOM) crash. Three things compete for that fixed budget:

1. Model weights — the model's size. 2. Activations — temporary numbers while processing. 3. KV cache — memory to "remember" the conversation during generation (Module C).

🅿️
Mental model: VRAM is a fixed-size parking lot. The model is a big bus always parked there; every active request parks more cars. Lot full → next car can't park → OOM.

A4 · Batch size — trading memory for speed

A batch is a group of requests processed together in one GPU pass. Bigger batch → more useful work per pass → higher throughput, but more VRAM used. Smaller batch → less VRAM, but the GPU is underused (low utilization). This is the central tension you'll debug: crank batch size for throughput → OOM crashes; keep it tiny → "the GPU isn't being used." Batch size is the dial connecting memory pressure and performance.

A5 · Training vs Fine-tuning vs Inference

Training = building a model from scratch (days/weeks, many GPUs, very hungry). Fine-tuning = nudging an already-trained model with a smaller specialized dataset (cheaper, but still "writes" a new model). Inference = using a finished model to answer requests — this is serving, and where most support work lives.

🎓
Analogy: Training = teaching a student everything. Fine-tuning = a specialist bootcamp for the graduate. Inference = the graduate now answering questions on the job.

Why it matters: their resource patterns differ. Training/fine-tuning are long-running batch jobs (HPC, Module D); inference is a long-lived service on live traffic (Kubernetes, Module B). Knowing which one a ticket involves instantly narrows your hypotheses.

A6 · Tensor vs Data parallelism (high level)

Data parallelism = a full copy of the model on each GPU, each serving different requests (like identical cashiers with their own lines) — used to handle more traffic. Tensor parallelism = the model is too big for one GPU, so the model itself is split across several GPUs working on one request together (cashiers shoulder-to-shoulder on one giant register) — used to fit giant models. When tensor parallelism is involved, the GPUs must talk to each other very fast (why GPU-to-GPU networking, Module F, matters).

A7 · Cold start latency

When an inference service first starts (or scales up a replica), it must load the model weights from storage into VRAM before answering anything. For a large model (tens of GB) this takes seconds to minutes. The first requests during this window are slow or fail — this is cold start. Once loaded ("warm"), responses are fast. This explains a huge number of "first request is slow after a deploy/restart" tickets, and ties into Module E (slow storage = slow load = long cold start).

Interview-readiness recap

▶QWhat happens when you run a model on GPU vs CPU?

The GPU does the model's massive parallel matrix math far faster than a CPU's few cores. The model weights must fit in the GPU's separate VRAM; the CPU usually handles orchestration, tokenization, and feeding data to the GPU.

▶QWhat is VRAM and what causes OOM?

VRAM is the GPU's own fixed memory. OOM happens when model weights + activations + KV cache exceed it — commonly from too-large batch size, too-big a model for the GPU, too many replicas sharing one GPU, or a memory leak.

▶QTraining vs fine-tuning vs inference?

Build from scratch / specialize an existing model / use a finished model to serve requests. Support work is mostly inference.

▶QWhy does batching improve throughput?

The GPU processes many requests in one parallel pass, keeping its thousands of cores busy, instead of wasting them on one request at a time.

▶QTensor vs data parallelism?

Data = full model copies for scale; tensor = one model split across GPUs because it's too big to fit on one.

▶QWhy cold start latency?

A fresh replica must load large weights from storage into VRAM before serving; first requests are slow until it's warm.

B

Kubernetes for AI Workloads

Your existing skill, reframed for GPUs
Why it matters
You already know Kubernetes — this module reframes it for GPU/AI workloads, the exact lens Together AI cares about. The job is mostly "workloads are misbehaving inside a cluster — diagnose it."

Concepts

B1 · Core objects (fast refresher)

Cluster = machines managed together; the control plane is the brain, worker nodes run workloads. Pod = smallest run unit (where your model server runs). Deployment = manages identical pods (replicas), rolling updates, self-healing. Service = stable network address + load balancing in front of changing pods (ClusterIP=internal, LoadBalancer=external). Namespace = a logical folder.

kubectl get pods -A           # what's running, what's broken
kubectl describe pod <pod>    # WHY a pod is stuck/pending (read Events at bottom)
kubectl logs <pod>            # what the app said
kubectl logs <pod> --previous # logs from the crashed previous run
kubectl get nodes             # are nodes Ready
kubectl top nodes / top pods  # CPU/memory usage
🔎
Rule of thumb: describe tells you why the scheduler/cluster is unhappy; logs tells you why the app is unhappy.

B2 · Requests vs Limits

requests = the guaranteed minimum (the scheduler uses it to find a node). limits = the maximum allowed — exceed the memory limit and the pod is killed (OOMKilled). Models are memory-hungry: limit too low → OOMKilled on load; requests too high → scheduler finds no node → pod stays Pending.

B3 · How K8s schedules GPUs (the AI-specific core)

Kubernetes can't see GPUs by itself. The pieces that make it work: a GPU device plugin (usually a DaemonSet) that advertises the node's GPUs as a schedulable resource nvidia.com/gpu; and a DaemonSet that runs one copy per node to deploy drivers/plugin everywhere.

resources:
  limits:
    nvidia.com/gpu: 1     # give this pod 1 GPU

GPUs are requested as whole units. If the device plugin isn't running, K8s thinks there are zero GPUs even on a GPU machine. If a pod requests a GPU and none is free, it stays Pending.

B4 · Keeping GPU nodes for GPU work

Taint = a "keep out" sign on a node (no pod schedules there by default). Toleration = a "permission slip" on a pod that lets it tolerate that taint. Node affinity = rules that attract a pod to certain nodes (e.g., only gpu-type=a100). So GPU nodes get a taint, GPU pods carry the matching toleration → only GPU pods land there.

🚪
Analogy: a taint is a velvet rope; a toleration is being on the guest list; node affinity is "I want the table by the window."

B5 · Pod pressure and eviction

When a node runs low on memory/disk, the kubelet starts evicting pods (often the ones using more than they requested). So "pods randomly evicted" usually means node memory or disk pressure, not a pod bug. Check kubectl describe node for MemoryPressure/DiskPressure.

B6 · Cluster autoscaling

Cluster Autoscaler adds nodes when pods can't schedule (e.g., Pending GPU pods) and removes idle ones. Horizontal Pod Autoscaler (HPA) adds replicas under load. The GPU gotcha: spinning up a GPU node is slow and pricey, and a new replica still has to cold start before it helps — so scaling up doesn't fix latency instantly.

B7 · The four failure states to know on sight

Pending → not scheduled: no node with enough CPU/mem/GPU, unmatched taints, or unsatisfiable affinity → describe pod. ImagePullBackOff → can't download the image: wrong name/tag, registry auth, network → describe pod. CrashLoopBackOff → starts then crashes repeatedly: app error, bad config/env, missing dependency, model load failure, or OOM → logs --previous. OOMKilled → exceeded memory limit: limit too low or using too much → describe pod then fix limits/reduce memory.

Interview-readiness recap

▶QA GPU pod is stuck Pending — how do you debug?

kubectl describe pod and read Events. Check: are there GPU nodes with free GPUs? Is the device plugin running? Does the pod's toleration match the node's taint? Does affinity match a real node? Is there a GPU quota limit?

▶QWhat causes CrashLoopBackOff?

The container keeps starting and dying — app crash, bad config/env, missing dependency, failed model load, or OOM. Check kubectl logs <pod> --previous.

▶Qlimits vs requests?

Requests = guaranteed minimum the scheduler reserves; limits = max allowed. Exceeding the memory limit → OOMKilled.

▶QHow does K8s schedule GPU workloads?

A device plugin advertises nvidia.com/gpu; pods request whole GPUs; if none free, they stay Pending. GPU nodes are protected with taints, and pods carry matching tolerations + affinity.

▶QPods being evicted randomly?

Usually node memory/disk pressure; the kubelet evicts pods (often over-request ones). Check describe node for pressure conditions.

▶QWorks in dev, fails in prod GPU cluster?

Compare GPU availability/quota, driver & CUDA version on nodes, taints/affinity, resource limits, image/registry access, and config/secrets differences.

C

Distributed AI Serving / Inference

The differentiator — reason about why serving is slow
Why it matters
Lots of candidates know Kubernetes; few can reason about why an inference service is slow or inconsistent. This turns "the pod looks fine but the customer says it's slow" into a real diagnosis.

Concepts

C1 · What serving / inference is

Inference serving = a long-lived service that holds a model in GPU memory and answers requests over an API. The software is a model server / inference engine. Two names to know: vLLM (high-performance open-source LLM engine, famous for PagedAttention + continuous batching) and Triton (NVIDIA's general-purpose model server with dynamic batching). Key insight: the model server decides how requests are batched, queued, and run on the GPU — many "performance" tickets are really server config issues.

C2 · Latency vs Throughput

Latency = how long one request takes (watch percentiles: p50 median, p95/p99 the slow tail users feel). Throughput = requests/tokens handled per second. They trade off: batching many requests is efficient for the GPU (high throughput) but makes each request wait for its batch-mates (higher latency). Almost every tuning decision is a latency-vs-throughput trade. First clarifying question to a customer: which one are they hurting on?

C3 · Request batching

Static batching = wait for N requests (or a timeout), then run together (first request waits). Continuous/dynamic batching (vLLM) = continuously add/remove requests from the running batch so the GPU never idles — the modern, efficient approach. When throughput is low or GPU utilization is low despite traffic, batching misconfigured or disabled is your top hypothesis.

C4 · The KV cache

An LLM generates one token at a time and must "remember" everything so far. To avoid recomputing, it stores intermediate results in the KV cache in VRAM. Crucially, the KV cache grows with prompt + output length. You can serve many short requests fine but a few very long ones cause OOM — the main cause of "OOM only on long requests." Engines like vLLM optimize exactly this.

C5 · Queueing & saturation

A GPU can only do so much at once; extra requests wait in a queue. If arrival rate > service rate, the queue grows and wait time adds to latency — so latency spikes even though the GPU is healthy and busy. This is the "GPU at 95%, latency exploding" pattern: the GPU isn't broken, it's saturated. Fix = more capacity, better batching, smaller requests, or load shedding — not "restart the pod."

📊
Familiar from Kafka: consumer lag when producers outpace consumers is the same shape — backlog from a throughput mismatch.

C6 · Streaming inference

Streaming sends tokens back as generated (answer appears word-by-word) instead of waiting for the whole response. It improves perceived latency even if total time is similar. Streaming tickets usually involve the connection (proxies/load balancers buffering the stream) rather than the model — a link to Module F.

C7 · Warm/cold start & replica consistency

Cold start (recap): a fresh replica must load the model before it's useful. Inconsistent responses across requests is a classic distributed-serving symptom: different replicas may run different model versions (a bad/partial rollout), different configs, or different sampling settings. "Same prompt, different quality" → suspect replica/version mismatch before the model.

C8 · Why huge models need distributed serving

If a model is too big for one GPU's VRAM, it's split across GPUs (tensor parallelism), which must exchange data during every request over a fast interconnect. So a network problem between GPUs shows up as an inference latency problem: in big-model serving, slow GPU-to-GPU networking looks like a slow model.

Interview-readiness recap

▶QAPI latency spikes from 200ms to 5s — what do you check?

Is traffic up (queue backlog/saturation)? GPU utilization (saturated vs idle)? Batching working? Request/output sizes (KV cache growth)? Cold start from a recent scale/deploy? Network/proxy buffering? A recent change? Narrow by isolated vs systemic and what changed.

▶QThroughput dropped 50% after scaling — why?

New replicas may be cold-starting; batching may not span replicas; a CPU/network/storage bottleneck now feeds GPUs poorly; or load balancing is uneven so one replica is saturated while others idle.

▶QResponses inconsistent across requests — why?

Replicas running different model versions/configs (bad rollout), different sampling settings, or cache differences. Check rollout status and version across replicas.

▶QWhat is the KV cache?

Per-request memory storing attention results so the model doesn't recompute; it grows with context length and is a common OOM cause on long requests.

▶QvLLM / Triton?

Model servers that handle batching/queueing/memory; vLLM is known for continuous batching + efficient KV-cache (PagedAttention).

Cheat-sheet

Latency = 1 request (watch p95/p99) Throughput = req/sec -> they TRADE OFF Slow + GPU ~95% + traffic up -> SATURATION/queue backlog (add capacity, not restart) Slow + GPU low util -> batching off, or CPU/net/storage bottleneck feeding GPU OOM only on long prompts -> KV cache growth (context too long / too many long reqs) Inconsistent answers -> replica/version/config mismatch (check rollout) Slow first reqs after deploy -> cold start (model loading) Streaming issues -> usually proxy/LB buffering, not the model Big model split on many GPUs -> slow GPU-GPU network LOOKS like slow inference

HPC / Cluster Concepts

The job-scheduler mindset
Why it matters
Together AI runs both inference services (K8s) and large batch jobs like training/fine-tuning (HPC). You need the support mindset for the HPC side: jobs, queues, schedulers, and why a job waits. Conceptual understanding is enough.

Concepts

D1 · HPC vs Kubernetes

HPC (High-Performance Computing) is big batch jobs on big clusters. The key difference: inference (K8s) = long-lived services on live traffic, scaled by replicas ("always-on restaurant"); HPC = jobs you submit, wait in line for, run to completion, and that then exit ("submit a print job to a shared printer queue").

D2 · Job schedulers & SLURM

A job scheduler decides which job runs where and when, by resources and priority. SLURM is the most common one. Workflow: submit a job ("4 GPUs for 6 hours, run this script") → it enters a queue → when resources free up SLURM allocates and runs it → it finishes/fails and releases resources.

sbatch   # submit a job
squeue   # see the queue / why a job is waiting
scancel  # cancel a job
sinfo    # node / partition status

D3 · Partitions, queues, priority

A partition is a named group of nodes with a purpose/policy (e.g., a gpu partition). A queue holds waiting jobs. A job waits because requested resources aren't free, priority is low, the partition is full, or it asked for more than the partition can ever provide (waits forever — a classic bug). Customer says "stuck in queue for hours"? Check: are the requested resources available in that partition? Is the request even satisfiable? Higher-priority jobs ahead? Partition full / nodes down?

D4 · Failures, retries, preemption

Job failure mid-run — a long job dies partway; without checkpointing all progress is lost. Preemption — a higher-priority job can kick out a running lower-priority one (paused/requeued), so "my job stopped" can be preemption, not a crash. Retry — schedulers can auto-resubmit failed jobs.

D5 · Compute vs storage separation

In big clusters, GPU (compute) nodes are separate from storage nodes; compute reads data/weights over the network from shared storage. The consequence (a frequent root cause): GPUs can be idle and healthy while the job is stuck waiting on slow storage — compute is starved because data isn't arriving fast enough. This bridges into Module E.

Interview-readiness recap

▶QWhat is a job scheduler / SLURM for?

It allocates cluster resources to submitted jobs by availability and priority, queuing jobs until their requested resources free up. SLURM is the standard HPC scheduler.

▶QA job is stuck in queue too long — what do you check?

Are the requested resources available in that partition? Is the request even satisfiable (not asking for more than exists)? Higher-priority jobs ahead? Partition full or nodes drained? Use squeue/sinfo.

▶QCompute nodes available but jobs aren't running — why?

The request may not match (wrong partition, more time/GPUs than allowed), the scheduler may hold for a reservation/priority, nodes may be drained, or a dependency/quota is blocking.

▶QA job fails mid-run — what happens?

It releases resources; progress is lost unless it checkpointed. With retries configured it may resubmit. Distinguish a crash from a preemption by a higher-priority job.

▶QGPUs idle but job stuck — why?

Often storage starvation — the GPUs are waiting for data/weights from slow shared storage (compute/storage separation).

Cheat-sheet

E

Storage Systems in AI Clusters

The quiet root cause behind many GPU incidents
Why it matters
Storage is the quiet root cause behind slow model loading, long cold starts, and idle-but-stuck GPUs. Models are huge files; getting them onto the GPU fast depends entirely on storage. Debugging this well makes you stand out.

Concepts

E1 · Why storage matters in AI

Two storage-dependent moments: model loading — weights (tens of GB) must be read from storage into the node and onto the GPU; slow storage = long cold start. Training/data feeding — training jobs continuously read large datasets; if storage can't keep up, GPUs starve. So "slow startup" and "idle GPUs" tickets often bottom out in storage.

E2 · NFS — the simplest shared storage

NFS (Network File System) lets many machines mount and read the same files over the network as if local. Simple and common, but a bottleneck under heavy parallel load: if dozens of GPU nodes all pull a big model from one NFS server at once, its network and disk get overwhelmed → everyone's load is slow. Fine at modest scale, strains under heavy AI load.

E3 · Distributed storage (VAST, WEKA)

Serious AI clusters use distributed/parallel file systems built for high bandwidth and many simultaneous readers. Two names: VAST and WEKA. The concept: instead of one server, data is spread across many storage nodes, so many GPU nodes can read in parallel at very high aggregate bandwidth — this is what keeps large GPU fleets fed.

E4 · The three storage symptoms

Bandwidth bottleneck — demand exceeds storage throughput; everything storage-dependent slows together, especially during mass scale-ups. File access latency — each read has high delay (far/overloaded storage); many small reads crawl. Model loading bottleneck — slow cold starts because weights load slowly. Mitigations: local caching of weights, faster/distributed storage, sharded model formats, pre-warming replicas.

E5 · "Fails to load" vs "loads slowly"

Loads slowly → bandwidth/latency/NFS contention (performance problem). Fails to load → a correctness problem: corrupted/incomplete file, missing file/dependency, wrong permissions, or the node ran out of disk while downloading. Different branch — read the exact error and verify file/permissions/disk.

Interview-readiness recap

▶QModel loads slowly after a restart — why?

Cold start reading large weights from storage. Suspect NFS/storage contention (many nodes pulling at once), bandwidth/latency limits, no local cache, or a far/overloaded backend. Mitigate with caching, faster/distributed storage (WEKA/VAST), or pre-warming.

▶QGPU nodes idle but jobs stuck waiting — why?

Storage starvation: compute is waiting on data/weights from slow shared storage (compute/storage separation). Check storage throughput/latency and mount health.

▶QWhat is NFS and why can it be a problem?

Shared network file system; simple but a single server can be overwhelmed when many GPU nodes read big models simultaneously, throttling everyone.

▶QModel fails to load (not slow) — what do you check?

Corrupted/incomplete file, missing dependency, wrong permissions, or out-of-disk on the node. Read the exact error in logs.

F

Networking

Point at the right layer, fast
Why it matters
A support engineer lives in networking: API timeouts, pods that can't talk, DNS failures, region-specific slowness, load-balancer misconfigs. You need a clean model of what happens when a request travels so you can point at the right layer fast.

Concepts

F1 · The life of an API request

When a client calls an inference endpoint: 1) DNS resolution (hostname → IP; if DNS fails, nothing connects). 2) TCP handshake (SYN, SYN-ACK, ACK; reliable transport under HTTP). 3) TLS handshake (encryption for HTTPS). 4) HTTP request (method, headers, JSON body). 5) Routing through a load balancer / ingress to a healthy pod. 6) Processing (Modules A/C). 7) HTTP response (possibly streamed). When something is slow/timing out, figure out which step is the culprit — that one discipline solves most network tickets.

F2 · TCP and HTTP basics

TCP = reliable, ordered byte delivery; a handshake opens the connection. A failure here means the client can't even reach the server (firewall, wrong port, server down). HTTP status codes tell you a lot: 2xx success; 4xx client error (401/403 auth, 429 rate-limited); 5xx server error (500 crash, 502/503/504 bad gateway / unavailable / gateway timeout). A 504 screams "the proxy waited for the backend and gave up" → backend overload/saturation (link to C5).

F3 · Kubernetes networking essentials

Pod IP vs Service IP — pods are ephemeral (IPs change); a Service gives a stable virtual IP + DNS name and load-balances. Always talk to the Service, not a pod IP. CNI (Container Network Interface) wires up pod networking; "pods can't talk" → suspect CNI or NetworkPolicies (firewall rules between pods). CoreDNS resolves Service names in-cluster; failed service-name lookups → suspect DNS. Ingress routes external HTTP by host/path to Services (via an ingress controller like NGINX); LoadBalancer exposes a Service externally.

F4 · Symptoms to triage

API timeout → walk the request life: DNS resolving? endpoint reachable (TCP)? Is it a 504 (backend overloaded) vs no connection (firewall/security group/DNS)? Use curl -v, nslookup. Pods can't communicate → Service vs pod IP, NetworkPolicies, CNI health, DNS; test from inside a pod. High latency → network distance/region routing, overloaded LB, or (common) backend saturation; confirm whether time is in transit or processing. Region-specific failures → per-region DNS/routing, a regional LB/endpoint down, firewall differences, or capacity only in some regions.

F5 · "Is it network or the app?"

The most valuable habit: determine where the time goes. curl -v + timing flags show connect time vs time-to-first-byte vs total. Connecting instant but first byte takes 5s → backend (app/GPU/queue), not the network. Connecting itself hangs → DNS/TCP/firewall. This stops you chasing the wrong layer.

G3 · A latency-measuring script (the classic ask)

import requests, time, statistics

URL = "https://api.example.com/v1/completions"
HEADERS = {"Authorization": "Bearer TOKEN", "Content-Type": "application/json"}
PAYLOAD = {"model": "my-model", "prompt": "test", "max_tokens": 16}

latencies, errors = [], 0
for i in range(100):
    start = time.time()
    try:
        r = requests.post(URL, json=PAYLOAD, headers=HEADERS, timeout=30)
        r.raise_for_status()
        latencies.append(time.time() - start)
    except Exception as e:
        errors += 1
        print(f"request {i} failed: {e}")

if latencies:
    latencies.sort()
    print(f"count={len(latencies)} errors={errors}")
    print(f"avg={statistics.mean(latencies):.3f}s")
    print(f"p50={latencies[len(latencies)//2]:.3f}s")
    print(f"p95={latencies[int(len(latencies)*0.95)]:.3f}s")

Talking points while writing: measure percentiles (p95) not just average (the tail is what customers feel — Module C2), count errors separately, and set a timeout so a hung request doesn't freeze your script.

G4 · A retry-with-backoff script

import requests, time

def call_with_retry(url, payload, headers, retries=3, backoff=1.0):
    for attempt in range(1, retries + 1):
        try:
            r = requests.post(url, json=payload, headers=headers, timeout=30)
            if r.status_code < 500:        # 2xx/4xx -> don't retry client errors
                return r
            print(f"attempt {attempt}: server {r.status_code}, retrying")
        except requests.RequestException as e:
            print(f"attempt {attempt}: {e}, retrying")
        time.sleep(backoff * 2 ** (attempt - 1))   # exponential backoff
    raise RuntimeError("all retries failed")

Key insight to say out loud: retry 5xx/network errors, but NOT 4xx (a 400/401 won't fix itself), and use exponential backoff so you don't hammer a struggling server (which worsens saturation — Module C5).

Observability & Debugging

Your home turf — lead with this strength
Why it matters

Concepts

H1 · Logs vs Metrics vs Traces

Logs = discrete event records ("request failed: CUDA OOM at 10:42") — best for what exactly happened. Metrics = numbers over time (GPU util %, latency p95, queue length) — best for trends, alerting, "is it healthy right now?" Traces = the path of a single request across components with per-hop timing — best for "where did the time go?" Metrics tell you something is wrong and when; traces tell you where; logs tell you why.

H2 · The GPU/inference metrics that matter

On top of CPU/mem/disk: GPU utilization (%) — low with traffic = the GPU is starved (batching off, or CPU/net/storage bottleneck); high (90%+) with rising latency = saturation. GPU memory (VRAM) — climbing toward the limit predicts OOM; a steady climb that never drops suggests a leak. Latency p50/p95/p99 — the customer-felt numbers. Queue length — backlog; the early warning before timeouts. Throughput and error rate (by type). The key skill: low vs high GPU utilization points you in opposite directions — low + slow = feeding bottleneck; high + slow = the GPU is the bottleneck (need capacity).

H3 · The tools

Prometheus collects/stores time-series metrics (query with PromQL; the DCGM exporter feeds GPU metrics in — good to name-drop). Grafana = dashboards + alerts on top. Datadog = all-in-one metrics + logs + traces. kubectl logs / kubectl top = quick no-dashboard checks.

H4 · A debugging method that ties it all together

Work the funnel (the senior mindset they're hiring for): 1) Scope it — isolated (one user/pod/node) or systemic? 2) What changed? — recent deploy, config, traffic spike, scale event? 3) Which layer? — compute, scheduling, network, storage, or app? 4) Confirm with the right signal — metrics to localize, traces to find the slow hop, logs to read the actual error. 5) Mitigate then RCA — stop the bleeding for the customer first (scale up, roll back, reroute, shed load), then root-cause.

H5 · Debugging with limited signal

A realistic curveball: "you only have logs — debug it." Reconstruct the timeline (when did errors start?), group errors by type/frequency (Module G's jq), correlate with any change, and form a hypothesis you can test (reproduce with curl). Show you can still reason without a perfect dashboard.

LLM Behaviour, Diagnostics & Error Codes

Bug vs normal vs platform — the day job
Why it matters
This is the module closest to the actual day job at Together AI. Customers call the inference API, something misbehaves, and you have to tell them whether it's their request, the model's nature, or the platform — fast. Half of support is knowing what an error code really means and which side of the fence the fix lives on. The other half is knowing which "bugs" are actually just how LLMs behave.

Concepts

I1 · How an LLM actually behaves (so you can tell "bug" from "normal")

LLMs are probabilistic next-token predictors, not deterministic functions. Internalize these behaviours or you'll chase non-bugs:

  • Non-determinism — the same prompt can give different answers across calls. This is expected when sampling is on (temperature > 0). "I get different outputs each time" is usually not a bug (see Module K for the knobs).
  • Hallucination — the model can produce fluent, confident, wrong content. It's a property of the technology, not a platform fault. You can't "fix" it in support; you mitigate with prompting, grounding/RAG, and lower temperature.
  • Context is finite — the model only "sees" what's in the current context window. It has no memory between calls unless the caller resends history. "It forgot what I told it earlier" = the client isn't sending prior turns.
  • Prompt-sensitivity — small wording/format changes shift output a lot. A customer whose quality "suddenly dropped" often changed their prompt or system message.
  • Repetition / degeneration — models can loop or ramble, especially at high temperature or with no repetition penalty, or when pushed past what they handle well.
  • Refusals — safety-tuned models decline some requests. A refusal is model behaviour, not an API error (still a 200 OK).

The support reflex: a valid-but-disappointing response is a model/prompt issue (HTTP 200); a failed response is an API/platform issue (non-200). That single split routes most tickets.

I2 · The inference infrastructure, end to end

When a customer request hits Together AI, it roughly travels: API gateway/auth → rate limiter → router (which model, which endpoint) → queue → inference server (e.g. vLLM) on a GPU node → model (weights in VRAM) → response (often streamed back). Two deployment shapes matter (detailed in Module K):

  • Serverless / shared — many customers share pooled capacity for a model. Elastic, pay-per-token, but subject to shared load (noisy neighbours, queueing under spikes).
  • Dedicated endpoints — capacity reserved for one customer. Predictable performance, no noisy neighbours, but you pay for idle and must size it.

Knowing the path means you can localize: auth problem (front), 429 (rate limiter), slow first token (queue/cold start), slow generation (decode/GPU), truncated output (token limits), dropped stream (network/proxy).

I3 · The diagnostic framework — five questions, in order

For any inference ticket, ask these in sequence. It mirrors the incident funnel (Module H) but tuned for the API:

  1. Is it a failed call or a bad response? Non-200 → platform/request. 200 but unhappy → model/prompt.
  2. What's the exact status code + error body? The body usually names the cause (see I4). Always get it.
  3. Isolated or systemic? One API key/model/region, or everything? Reproduce with curl.
  4. What changed? New prompt, longer inputs, higher traffic, a model version/deprecation, new SDK.
  5. Where in the path? Auth → rate limit → queue/cold start → generation → streaming/network.

I4 · Error codes — the ones you'll actually see, and what they mean

HTTP status tells you which side owns the fix. 4xx = the caller's request; 5xx = the server/platform. The high-value ones for an inference API:

  • 400 Bad Request — malformed request: bad JSON, wrong field, invalid parameter combo (e.g. temperature out of range). Caller fix.
  • 401 Unauthorized — missing/invalid/expired API key. Check the Authorization: Bearer header. Caller fix.
  • 403 Forbidden — authenticated but not allowed: no access to that model, org/billing issue, region restriction. Caller/account fix.
  • 404 Not Found — usually a wrong or deprecated model name, or wrong endpoint path. Very common — model IDs change and get retired. Caller fix (update model name).
  • 413 / "context length exceeded" (often 400/422) — input + requested max_tokens exceeds the model's context window. Extremely common. Fix: shorten prompt, lower max_tokens, or use a longer-context model. Caller fix (but you diagnose it).
  • 422 Unprocessable Entity — request is well-formed but semantically invalid (e.g. a parameter the model doesn't support). Caller fix.
  • 429 Too Many Requests — rate limit hit (requests-per-minute or tokens-per-minute) or, on shared capacity, model overloaded. Respect Retry-After, back off with jitter, or request a higher limit / move to a dedicated endpoint. Caller mitigates; platform may need capacity.
  • 500 Internal Server Error — something broke server-side. Usually transient; retry with backoff. If persistent, it's a platform incident. Platform fix.
  • 502 Bad Gateway / 503 Service Unavailable — an upstream/inference server is unreachable or overloaded, or capacity is scaling. Often transient; retry. Platform fix.
  • 504 Gateway Timeout — the backend didn't respond in time: a very long generation, an overloaded/saturated model, or a cold start. Reduce max_tokens, stream, or retry. Platform + caller.

Memory hook: 401/403 = keys/permissions, 404 = model name, 429 = slow down, 400/422/context = fix the request, 5xx = our side.

I5 · Troubleshooting playbooks (symptom → likely cause → action)

  • "All my calls return 401" → key wrong/expired/not sent → verify the Authorization header and that the key is active; reproduce with curl.
  • "404 model not found" → wrong or deprecated model ID → confirm the exact current model name from the model list; check deprecation notices.
  • "Getting 429s" → rate limit or overload → check RPM/TPM vs their traffic; add backoff + jitter; batch or spread load; raise limits or move to dedicated.
  • "context length exceeded" → prompt + max_tokens > window → count tokens; trim history/prompt; lower max_tokens; or switch to a longer-context model.
  • "Responses are truncated / cut off mid-sentence" → hit max_tokens (or the model emitted a stop sequence) → raise max_tokens, check finish_reason (length vs stop), check stop settings.
  • "First token is very slow, then it's fine" → cold start / queue on shared capacity → expect it after idle; dedicated endpoint or keep-warm helps.
  • "Generation itself is slow" → long output (decode), big model, or saturation → smaller/faster model, stream, reduce max_tokens, or dedicated capacity.
  • "Streaming just stops / connection drops" → proxy/load-balancer buffering or client timeout, not the model → check client read timeout, disable proxy buffering, confirm SSE handling.
  • "Outputs changed / got worse with no code change on our side" → prompt change, longer inputs, sampling settings, or a model version update/deprecation → diff their request; check model version; pin a version if available.
  • "Same prompt, different answers" → sampling on (temperature > 0) → this is normal; set temperature=0 (or low) + fixed seed if supported for reproducibility.
  • "Empty response / just whitespace" → stop sequence triggered immediately, max_tokens too low, or a bad chat template → check finish_reason, max_tokens, and prompt formatting (Module J).

Chat Templates

How messages become the string a model expects
Why it matters
This is a genuinely under-appreciated source of "the model is dumb" tickets — and a great topic to sound sharp on. Chat models are trained to expect a very specific text format with special tokens marking who said what. If the prompt is assembled wrong, the model gets confused, ignores the system prompt, rambles, or won't stop. When a customer uses the raw completions endpoint or self-hosts, chat templates become their responsibility — and yours to debug.

Concepts

J1 · What a chat template actually is

Under the hood, a chat model doesn't see "roles" like system/user/assistant. It sees one flat string of tokens. A chat template is the rule that turns a structured conversation (a list of role+content messages) into that exact string, inserting the special tokens the model was trained on to mark turn boundaries.

Two levels to keep straight:

  • The Chat Completions API (/v1/chat/completions) — you send structured messages ([{"role":"system",...},{"role":"user",...}]) and the server applies the correct chat template for you. This is the easy, recommended path; most template bugs disappear here.
  • The raw Completions / self-hosted path — you send a single prompt string and you must format it with the model's template yourself. Get it wrong and quality tanks. This is where template tickets come from.

J2 · Why the format is so strict

Each model family was fine-tuned on its own template. The special tokens (turn markers, role headers, end-of-turn markers) are how the model learned "this is where the user's turn ends and mine begins." If you:

  • use the wrong family's template → the model sees unfamiliar tokens and behaves erratically;
  • forget the end-of-turn token → the model may not know when to stop (runs on, or answers as the user);
  • skip the assistant-turn header → the model may not realize it's its turn to speak;
  • duplicate or mis-nest tokens → confusion, ignored system prompt.

Different families use different schemes — Llama 3 uses <|begin_of_text|> and <|start_header_id|>role<|end_header_id|> with <|eot_id|>; ChatML (used by Qwen and others) uses <|im_start|>role ... <|im_end|>; Mistral/Mixtral use [INST] ... [/INST]. You don't need to memorize every one — you need to know they differ per model and must match exactly.

J3 · The system / user / assistant roles

  • system — instructions and persona ("You are a helpful assistant that answers in JSON"). Sets behaviour for the whole conversation.
  • user — the human's messages.
  • assistant — the model's previous replies (you resend these to give it memory of the conversation).

For multi-turn chat, the client resends the whole message list each call (system + all prior user/assistant turns + new user turn). This is why "it forgot" happens — the client didn't include history (link to Module I1).

J4 · Common chat-template failures (and the fix)

  • Model won't stop / keeps generating / answers as the user → missing end-of-turn token or missing assistant header in a hand-built prompt → use the Chat Completions API, or apply the model's exact template (and set correct stop tokens).
  • System prompt ignored → template put the system message somewhere the model doesn't expect, or the model family handles system prompts differently → verify the template; some models fold system into the first user turn.
  • Garbled / erratic output on a self-hosted model → wrong template for that model → match the template to the exact model.
  • Special tokens showing up as literal text in the output → tokenization/template mismatch → confirm the tokenizer and template are the model's own.
  • Tool/function calling not working → tool-calling has its own template/format per model → follow that model's tool-format spec.

The golden rule to say out loud: prefer the Chat Completions API so the server owns the template; only hand-format when self-hosting, and then match the model's template exactly.

Model Performance, Deployment & Config

The knobs, and what each one breaks
Why it matters
When a customer asks "why is it slow / expensive / low quality, and what should I change?", this is the module that answers. It's the knobs (sampling + token params), the deployment choices (serverless vs dedicated), and the optimizations (quantization, context length, parallelism) — the levers you'll actually recommend in tickets and in the Forge scenario.

Concepts

K1 · The generation parameters (the knobs customers get wrong)

These control output behaviour, not speed of the hardware. Know what each does and its failure mode:

  • temperature — randomness. 0 = (near) deterministic, focused; higher (0.7–1.0+) = more creative/varied. Too high → rambling, off-topic, hallucination-prone. "Answers are inconsistent" → lower it. "Answers are robotic/repetitive" → raise it slightly.
  • top_p (nucleus sampling) — restrict choices to the smallest set of tokens whose probability sums to p (e.g. 0.9). Another diversity control; usually tune temperature or top_p, not both hard.
  • top_k — restrict to the k most likely tokens. Similar purpose.
  • max_tokens — cap on output length. Too low → truncated answers (finish_reason: length). Too high → slower and more expensive, and risks exceeding the context window.
  • stop / stop sequences — strings that force generation to end. Misused → empty or cut-off output.
  • repetition / frequency / presence penalty — discourage repeating tokens. Helps with loops/ramble; too high → unnatural, avoids necessary words.
  • seed — if supported, fixes the random draw so results are reproducible (pair with temperature=0).

Support reflex: quality/consistency complaints usually trace to these params before any infrastructure.

K2 · Latency anatomy (what "slow" actually means)

Split every latency complaint into two numbers (ties to Module A/C):

  • Time to first token (TTFT) — how long until the first token appears. Driven by prefill (processing the input), queueing, and cold start. Long prompt or cold/queued endpoint → slow TTFT.
  • Time per output token / total generation time — driven by decode, one token at a time. Long outputs and bigger models → slower overall.

So: long input → slow first token; long output → slow overall. The fixes differ — shorten prompt vs cap max_tokens/pick a faster model. Streaming hides TTFT-plus-decode by showing tokens as they come; it doesn't make total time shorter but massively improves perceived speed.

K3 · Serverless vs Dedicated deployment

The core deployment decision you'll explain constantly:

  • Serverless / shared — pay per token, instant to start, elastic, no ops. But shared capacity means possible queueing under load, noisy neighbours, and cold starts after idle. Best for spiky/low/medium or getting started.
  • Dedicated endpoint — reserved GPUs for one customer. Consistent low latency, no noisy neighbours, higher and predictable throughput, ability to keep the model warm. You pay for reserved capacity (including idle) and must size it. Best for steady high volume, strict latency SLAs, or predictable cost at scale.

The recommendation logic: spiky/uncertain/low volume → serverless; steady high volume or latency-sensitive → dedicated. This is the exact tradeoff (opex-per-token vs reserved-capacity) from the token-economics discussion — and a very likely Forge prompt.

K4 · Quantization (the big performance/cost lever)

Quantization stores model weights at lower numeric precision — FP16/BF16 (16-bit) → FP8 (8-bit) → INT4 (4-bit) and formats like AWQ/GPTQ. Effects:

  • Less VRAM — a smaller footprint means the model fits on fewer/smaller GPUs and leaves more room for KV cache (bigger batches, longer context).
  • Faster + cheaper — less data to move and compute, so higher throughput / lower cost.
  • Small quality cost — aggressive quantization (INT4) can slightly degrade quality; well-done 8-bit is usually near-lossless.

Support relevance: quantization is a lever for OOM ("use a quantized version to fit"), for cost/throughput, and it explains why "the same model" can behave slightly differently across providers/precisions.

K5 · Context length & KV cache in deployment

The context window (max input + output tokens) is a per-model, per-deployment property. Longer context = more capability but more VRAM for the KV cache and higher latency/cost (link to Module C4). Serving choices — max context, max batch size, KV-cache memory — trade off how many concurrent/long requests a deployment can handle before OOM or queueing. "We can't send long documents" or "long requests OOM the endpoint" lands here.

K6 · Scaling & parallelism knobs (deployment config)

  • Replicas / autoscaling — more replicas = more concurrent throughput; new replicas cold-start (Module B6).
  • Tensor parallelism (TP) — split one big model across GPUs so it fits / runs faster; needs fast GPU-GPU interconnect (Module A6/C8).
  • Max batch size / concurrency — higher = more throughput but more VRAM and potentially higher per-request latency (the latency-vs-throughput trade, Module C2).
  • Keep-warm / min replicas — avoids cold starts for latency-sensitive dedicated endpoints, at the cost of paying for idle.

K7 · The optimization playbook (what you recommend)

When a customer wants faster / cheaper / better, reach for these in order:

  • Faster/cheaper: smaller or quantized model → cap max_tokens → stream → right-size or dedicate capacity → increase batch/concurrency (throughput) → shorten prompts.
  • More consistent/higher quality: lower temperature (+ seed) → tighten the prompt/system message → add repetition penalty for loops → move up a model tier only if needed → ground with retrieval for factuality.
  • Fitting bigger models / longer context: quantize → tensor parallelism → longer-context model → manage KV-cache/batch settings.

nvidia-smi: Reading the GPU

The first command you run on a GPU node
Why it matters
When a customer says "the GPU is slow / crashing / not being used," nvidia-smi is the very first command you run on the node to see ground truth. It talks directly to the NVIDIA driver and GPU and answers the questions every other module raises: Is the GPU even visible? How full is VRAM? Is it actually busy? What's running on it? Is it too hot or power-throttled? If you can read an nvidia-smi table out loud and say what's wrong, you've proven the single most practical GPU-support skill. Expect the interviewer to paste an output and ask "what do you see?"

Concepts

M1 · What nvidia-smi is

nvidia-smi (NVIDIA System Management Interface, also called NVSMI) is a command-line tool for monitoring, managing, and diagnosing NVIDIA GPUs (Tesla, Quadro, GRID, GeForce). It ships with the driver and can monitor performance/temperature/utilization, manage power/clocks/ECC, control persistence and compute modes, and emit machine-readable metrics for automation. Mental model: it's top (or kubectl get) but for the GPU.

M2 · Reading the default output (this is the interview question)

Running plain nvidia-smi prints two tables. Table 1 describes each GPU; Table 2 lists the processes using them.

Table 1 — the fields you must be able to interpret on sight:

  • GPU / Name / Bus-Id — the GPU index (0,1,2… — matters on multi-GPU nodes), model name, and PCI address (domain:bus:device.function) used to target one device.
  • Temp — core temperature in °C. Normal is fine; ~90 °C+ is a red flag for thermal throttling. (In a managed cloud the datacenter handles cooling, but you still watch it.)
  • Perf — performance state P0 (max) → P12 (min). A GPU that should be working but sits at a low-power P-state is a clue.
  • Persistence-M — persistence mode On/Off. On keeps the driver loaded even with no active client, which cuts startup latency for CUDA apps — relevant to cold-start/first-request-slow tickets.
  • Pwr:Usage/Cap — current power draw vs the cap, in Watts. Near the cap = possible power throttling.
  • Memory-Usage — VRAM used / total. This is the number for OOM debugging (Module A3). Note: frameworks like TensorFlow/Keras grab all VRAM on launch by default, so "memory full" doesn't always mean genuinely used — a classic false alarm.
  • GPU-Util — percent of time the GPU was running kernels over the sample window. This is the low-vs-high fork from Module H: low util with traffic = the GPU is starved (data/CPU/network/storage bottleneck feeding it, e.g., slow disk reads of mini-batches); high util = it's actually busy/saturated. Crucial caveat: GPU-Util is time-busy, not how fully the cores are used — it can read 100% while the GPU is only lightly loaded.
  • Volatile Uncorr. ECC — count of uncorrectable ECC memory errors since the driver loaded. Non-zero = suspect failing GPU memory (hardware).
  • Compute M. — compute mode (see M6); "Default" allows multiple processes to share the GPU.

Table 2 — per-process: GPU index, PID, Type (C=Compute, G=Graphics, C+G=both), Process Name, and GPU Memory Usage per process. This tells you which process owns the VRAM — essential when a leftover/zombie process is hogging memory and blocking a new model from loading.

M3 · The commands you'll actually use in support

General inspection:

nvidia-smi                 # the two-table summary (start here)
nvidia-smi -L              # list all GPUs with their UUIDs
nvidia-smi -q              # full detailed dump for all GPUs
nvidia-smi -i 0 -q         # detailed dump for GPU index 0 only
nvidia-smi -q -d ECC       # just one category (ECC, TEMPERATURE, PERFORMANCE, PCI, FAN...)
nvidia-smi topo -m         # GPU-to-GPU topology (PCIe/NVLink) — matters for tensor parallelism

Continuous / live monitoring (watch a problem as it happens):

nvidia-smi -l 5                              # refresh every 5 seconds
nvidia-smi -lms 500                          # refresh every 500 ms
nvidia-smi --filename=/var/log/gpu.log -l 5  # log to a file while looping
nvidia-smi dmon                              # scrolling device monitor (pwr, temp, sm%, mem%, clocks)
watch -n 2 nvidia-smi                        # simple alternative refresh

Process monitoring (find and kill the memory hog):

nvidia-smi pmon -c 1     # one sample of per-process GPU usage
nvidia-smi pmon          # continuous per-process monitor
sudo kill -9 <pid>       # terminate a stuck/leftover process holding VRAM

M4 · Scripting nvidia-smi (--query-gpu + --format=csv)

For dashboards, alerts, or a quick diagnostic script, use structured queries instead of parsing the pretty table.

# Pick exactly the fields you want, machine-readable:
nvidia-smi --query-gpu=index,name,temperature.gpu,utilization.gpu,memory.used,memory.total \
  --format=csv

# For scripts, strip headers and units so the numbers parse cleanly:
nvidia-smi --query-gpu=temperature.gpu,power.draw --format=csv,noheader,nounits

Common query fields worth knowing: identity (index,name,uuid,serial), performance (utilization.gpu,utilization.memory,pstate), memory (memory.total,memory.used,memory.free), temperature (temperature.gpu), power (power.draw,power.limit), driver (driver_version). Automation tips: prefer --format=csv,noheader,nounits, and use GPU UUIDs (not indexes) for stable identification across reboots.

A minimal logging loop you could sketch in an interview:

#!/bin/bash
LOG="/var/log/gpu_monitor.csv"
echo "timestamp,index,uuid,util,mem_used,mem_total,temp,power" > "$LOG"
while true; do
  nvidia-smi --query-gpu=timestamp,index,uuid,utilization.gpu,memory.used,memory.total,temperature.gpu,power.draw \
    --format=csv,noheader >> "$LOG"
  sleep 5
done

M5 · Management commands (know they exist; use with care)

These change GPU state and usually need sudo. You're unlikely to run them casually in prod, but interviewers like to hear you know them:

sudo nvidia-smi -pm 1                     # enable persistence mode (faster CUDA startup)
sudo nvidia-smi -pl 250                   # set power limit to 250 W

sudo nvidia-smi -i 0 --gpu-reset          # reset GPU 0 (only if no processes are using it)

M6 · ECC and Compute Mode

ECC (Error Correction Code) detects and corrects GPU-memory errors; check it with nvidia-smi -q -d ECC and watch the uncorrectable count — rising errors mean suspect hardware. Compute Mode controls sharing: 0 Default (multiple processes share the GPU), 1 Exclusive-thread, 2 Prohibited, 3 Exclusive-process; set with sudo nvidia-smi -c <mode>. If a customer says "only one job can use the GPU at a time," a non-Default compute mode is a prime suspect.

M7 · Common nvidia-smi failures and what they mean

  • "NVIDIA-SMI has failed because it couldn't communicate with the NVIDIA driver" → the driver isn't loaded / is mismatched — reload or reinstall it. This is the command-line face of Module A2's "driver/CUDA not healthy" branch, and inside K8s it maps to the device plugin seeing zero GPUs (Module B3).
  • Insufficient permissions → management commands need sudo.
  • GPU reset not supported → some cards/instances don't allow reset.
  • Compute-mode change didn't stick → may require a reboot to persist.
